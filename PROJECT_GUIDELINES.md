# Bankir Project Guidelines

Рабочий документ для дальнейшей доработки сайта `Bankir`.

Эти 5 правил считаются базовыми для всех следующих UI-изменений в проекте.

## 1. `prefers-reduced-motion`

Что внедряем:
- поддержка `@media (prefers-reduced-motion: reduce)`
- отключение или сильное сокращение несущественных анимаций
- отключение `smooth scroll` для пользователей с reduced motion

Почему:
- это бесплатное улучшение accessibility и UX
- снижает когнитивную нагрузку
- делает motion-систему проекта зрелой, а не декоративной

Правило:
- любая новая анимация или transition в проекте должна корректно деградировать в режиме reduced motion

## 2. Единые motion tokens

Что внедряем:
- единые easing-токены
- единые duration-токены
- системные transition-правила для hover, active, open/close состояний

Текущая база:
- `--ease-out-strong`
- `--ease-in-out-strong`
- `--duration-press`
- `--duration-fast`
- `--duration-ui`
- `--duration-open`

Рекомендуемые значения:
- `--ease-out-strong: cubic-bezier(0.23, 1, 0.32, 1)`
- `--ease-in-out-strong: cubic-bezier(0.77, 0, 0.175, 1)`
- `--duration-press: 140ms`
- `--duration-fast: 160ms`
- `--duration-ui: 200ms`
- `--duration-open: 240ms`

Почему:
- чтобы все состояния ощущались системно
- чтобы не появлялись случайные анимации с разным характером и скоростью

Правило:
- не писать случайные `200ms ease`, `300ms ease-in`, `transition: all`
- по умолчанию использовать существующие motion tokens
- для часто используемых элементов держать motion короче и тише, чем для раскрывающихся блоков

## 3. Легкий `:active` для CTA и pressable controls

Что внедряем:
- легкий `transform: scale(0.97)` для кнопок и нажатий
- tactile feedback для CTA и интерактивных контролов

Почему:
- улучшает tactile feel
- интерфейс ощущается отзывчивым
- нажатие визуально подтверждается мгновенно

Правило:
- все primary pressable элементы должны иметь аккуратный active-state
- active-state должен быть коротким, легким и не ломать layout

## 4. Аккуратный transition для FAQ open/close

Что внедряем:
- чистый open/close transition
- предсказуемую анимацию для текста ответа и иконки
- дешевые по производительности свойства: `opacity`, `transform`, `max-height`

Почему:
- текущие UI-переходы должны ощущаться чисто и без резкости
- FAQ — повторяемый паттерн, он должен быть сделан системно

Правило:
- для раскрывающихся блоков не использовать грубое переключение `display: none/block` как единственный механизм визуального состояния
- transition должен быть коротким, читаемым и не вызывать scroll jank

## 5. Ревизия тяжелых теней и фильтров

Что внедряем:
- контроль всех тяжелых `filter`, `drop-shadow`, `backdrop-filter`
- упрощение дорогих эффектов там, где они мешают scroll performance
- визуальный polish без потери плавности страницы

Почему:
- это нужно для окончательной чистоты скролла
- тяжелые эффекты быстро создают repaint/compositing проблемы

Правило:
- перед добавлением blur, filter, больших теней или сложных масок нужно проверять их влияние на scroll и rendering
- performance важнее декоративного эффекта, если эффект не дает критической пользы

## Как применяем эти правила в проекте

Любая следующая доработка интерфейса проверяется по 5 вопросам:

1. Есть ли корректное поведение в `prefers-reduced-motion`?
2. Используются ли существующие motion tokens?
3. Есть ли у интерактивного элемента адекватный `:active`?
4. Чисто ли работает open/close transition, если элемент раскрывается?
5. Не создает ли эффект лишнюю нагрузку на scroll и rendering?

## Область применения

Эти правила применяются ко всем следующим частям сайта:
- header
- hero
- scenario cards
- how it works
- FAQ
- CTA
- footer

## Стандарт для новых правок

Новые UI-правки должны быть:
- визуально точными
- системными по motion
- легкими по performance
- доступными по reduced motion
- безопасными для дальнейшей Figma-driven доработки
